DNA Cloud · design for review · adr-dna-cloud-saas

The hosted SaaS over DNA’s MCP runtime face.

Phase A hosts the DNA MCP server on managed cloud behind an IdP. Phase B is the SaaS on top: users sign up, pick a plan, register an OAuth client, and consume the DNA tools under a per-plan quota — billed by a provider. The load-bearing move: the token already carries a tenant claim; DNA Cloud adds a plan/quota claim through the same auth bridge, so the server enforces quota without rebuilding auth.

Enforcement — the reusable piece

Extend the Phase-2 bridge that already maps claim → tenant to also map claim → plan, then meter and limit at the one choke point. No auth is rebuilt — a resolver, a counter, a limiter.

STEP 01

Verify

FastMCP validates the OAuth 2.1 bearer token (PKCE, signature, audience) — exactly as Phase 2 ships it.

Authorization: Bearer eyJ…
STEP 02

Resolve

Bridge maps the token’s claims to a tenant and a plan. No plan claim → Free (never “unlimited”).

tenant: acme
plan: pro → Plan/pro
STEP 03

Meter

Every metered tool call is counted per (tenant, plan) in the daily quota window — one counter at the choke point.

count(acme,pro,2026-07-11)++
STEP 04

Limit

Check count + rate against the resolved caps and the plan’s feature families, then dispatch or deny.

6,240 / 10,000 → allow
✓ within quota & family unlocked → tool runs × over daily quota → 429 × payment failed / no plan → 402 × family not in plan (e.g. Free calls emit) → 403

Billing — flat + quota + overage

The server is already the meter, so it pushes usage records to the provider; the provider’s webhook flips the plan claim so enforcement always follows billing state.

07

Phasing — honest scope

The MVP is small because auth, tenancy and transport already shipped in Phase 2. What’s genuinely large is flagged.

SaaS MVP
  • Free + Pro only
  • Plan claim in the token
  • Quota + rate-limit on the existing bridge
  • Minimal portal — signup, pick plan, DCR, a usage number
  • One flat Pro subscription + overage on a provider sandbox
Enterprise (later)
  • Custom limits & SLA
  • SSO / org → tenant mapping
  • Private / VPC deploy
  • Invoicing & dedicated support
  • Sales-driven, not MVP
Big builds (flagged)
  • The portal web app itself (new repo, new surface)
  • Durable metering store — per-tenant counters, not in-process
  • Billing reconciliation + dunning
  • Abuse / DDoS protection on a public metered endpoint
08

Open decisions — for the owner

Six calls needed before build. Recommendations noted; the numbers and the open-core line are yours.

Separate repo for the portal?

Recommended: yes — a dna-cloud repo keeps the OSS SDK/CLI a pure library (the open-core boundary).

Billing provider?

Stripe-class assumed for flat+quota+overage & webhooks. Confirm the provider (keep it behind a thin port so it’s swappable).

Concrete tier limits & prices?

The calls/day, rate, tenant counts and the Pro flat price above are a starting proposal — set the real numbers.

Open-core split?

Recommended: SDK + CLI stay free/MIT; hosting + SaaS are the paid layer. Confirm the boundary is where you want it.

Billing model?

Recommended: flat + quota + overage (predictable bills) over pure per-invocation. Confirm.

MVP plan set?

Recommended: Free + Pro for the MVP, Enterprise deferred to a sales-driven phase. Confirm.
